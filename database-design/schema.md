# Database Schema Design

This document outlines the data schema specifically for the **AI Response Analysis** module implemented in TruthLens. The scope is strictly limited to hallucination detection, claim extraction, confidence scoring, and analysis history.

---

## 📌 System Collections

The TruthLens system utilizes a streamlined NoSQL schema. The primary entity is the **Analyses** collection, which natively embeds **Claims** and **Flagged Statements** directly within it to optimize read performance.

---

## 1. Analyses Collection

**Purpose:** Stores the complete result of a single AI text analysis, including the original response, internal score calculations, and historical tracking.

### Field Dictionary

| Field Name | Data Type | Description |
| :--- | :--- | :--- |
| `_id` | ObjectId | Unique identifier generated by MongoDB. |
| `originalResponse` | String | The raw AI-generated text submitted by the user. (Stored as `originalText` in code). |
| `confidenceScore` | Number | The calculated percentage score (0-100) indicating text reliability. |
| `confidenceLabel` | String | Categorical reliability tag (`HIGH`, `MID`, or `LOW`). |
| `claims` | Array (Embedded) | List of structured claims extracted from the text. |
| `flaggedStatements` | Array (Embedded) | Sentences marked for potential hallucinations with reasons. |
| `createdAt` | Date | Timestamp of when the analysis was performed (automatically generated). |
| `updatedAt` | Date | Timestamp of the last update to the record. |

### Embedded Claims Structure

Each object within the embedded `claims` array follows this sub-schema:

| Field Name | Data Type | Description |
| :--- | :--- | :--- |
| `_id` | ObjectId | Unique sub-document ID automatically assigned by MongoDB. |
| `text` | String | The specific factual, statistical, or historical claim extracted. |
| `type` | String | The categorization of the claim (`factual`, `statistical`, `historical`, `general`). |
| `confidence` | Number | The individual verifiable confidence weight of this specific claim. |

## 🔗 Relationships Summary

In a traditional relational SQL database, `Analyses` and `Claims` would be separate tables linked by an `analysisId` foreign key. However, in our optimized MongoDB implementation, **Claims are deeply embedded** into the Analysis document. There is a **One-to-Many** relationship (One Analysis has Many Claims), but it is resolved at the document boundary, ensuring that fetching an analysis instantly retrieves all associated claims without requiring slow `JOIN` operations.
